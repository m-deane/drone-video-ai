# Spec: drone-video-pipeline
Status: DRAFT

## Problem Statement

This repository currently contains no application code — it is an inherited copy of the `claude-template` maintainer source tree (root `CLAUDE.md`, `README.md`, `mkdocs.yml`, `docs/`, `promptlab/`, `copilot-studio/`, `assets/`, `dist/`, `site/`, `sync-claude-template.sh` all still describe/serve the template project itself, not a video pipeline) plus one genuinely project-relevant asset: `example-photos-videos/`, which holds raw sample footage and a `DJI_0355/manifest.json` produced by some external prior tool. That manifest already scores 15-second clips 0–100 on a `score`/`motion_energy`/`motion_type` (`REVEAL`, `ORBIT_CW`, `STATIC`, ...) taxonomy and carries `post_processing` fields (`color: "drone_aerial"`, `stabilize`, `letterbox`, `auto_speed`) — useful schema precedent, but also evidence that a prior tool conflated selection with pixel-level post-processing, which this project must explicitly avoid repeating in its stitching stage.

The goal is to convert this repo into a drone video editing AI pipeline with three capabilities, each independently valuable and independently shippable:

1. **Highlight extraction** — given raw, unedited drone footage, automatically split it into variable-length highlight snippets selected primarily on visual/technical quality (sharpness, exposure, motion smoothness/stabilization, scene boundaries, composition) rather than on subject-matter or manual review.
2. **Reel stitching** — given a set of highlight snippets and/or other supplied footage, assemble a final reel with controllable duration, pacing, and transition style, under a hard constraint that no frame is ever color-graded, corrected, or otherwise pixel-modified — only cut points, ordering, and transitions on unmodified source footage, with every decision recorded in an EDL/manifest.
3. **Reference pack** — curate metadata (and, only where license terms clearly permit, footage) of award-winning drone landscape/wildlife videos as training/eval exemplars, scored on the same quality metrics as capability 1, to give the project a working operational definition of "award-winning" quality it can measure against.

This spec covers all three capabilities as one initial spec artifact (per the research recommendation to eventually split into three specs, this first pass establishes shared scope/criteria; the Open Questions section flags the eventual split as one of the choices for sign-off).

## Scope (in)

**Capability 1 — Highlight extraction**
- A CLI/pipeline that ingests one or more raw drone video files and outputs: (a) a JSON manifest of candidate/selected segments with in/out timecodes and per-signal + composite quality scores, and (b) optionally, rendered clip files matching those segments.
- Segmentation on hard scene/shot boundaries (PySceneDetect `AdaptiveDetector`, chosen specifically for its resistance to false triggers during camera panning/movement) unioned with motion-derivative local minima (the point between two maneuvers, e.g. end of an orbit or top of a reveal) as additional candidate cut points, honoring a configurable min/max clip duration.
- Per-segment scoring on: sharpness (Laplacian variance via OpenCV `cv2.Laplacian(...).var()`), exposure (ffmpeg `signalstats` clipping/histogram stats or an equivalent NumPy/OpenCV histogram), motion smoothness (jerk/acceleration of a per-frame camera-motion transform time series, derived either from ffmpeg `vidstabdetect`'s first-pass log or a pure-OpenCV optical-flow reimplementation of the same technique — see Open Questions for the GPL-vs-permissive tradeoff between those two), and composition (saliency-to-rule-of-thirds distance via OpenCV's `saliency` module, plus a vendored Hough-line horizon-tilt scorer for aerial-specific framing).
- Hard-gate exclusion (not just down-weighting) of unusable segments via ffmpeg `blackdetect`/`freezedetect` (lens-cap, frozen-frame, sensor failure).
- A documented, versioned scoring-weight configuration (not hardcoded constants) so weights can be tuned per footage type without code changes.
- Reuse of the existing `DJI_0355/manifest.json` schema shape (`score`, `motion_energy`, `motion_type`, per-clip start/end/duration) as a compatibility/migration reference, extended with the new per-signal quality fields — but explicitly WITHOUT carrying forward its `post_processing` block's pixel-editing fields (`color`, `stabilize`, `letterbox`, `auto_speed`) into any manifest this pipeline produces for capability 2 input.

**Capability 2 — Reel stitching**
- A CLI/pipeline that ingests an ordered (or orderable) set of clips — highlight-extraction output and/or other supplied footage — plus pacing/duration/transition-style parameters, and produces a final stitched reel.
- A JSON edit manifest authored before any rendering as the source of truth: ordered `{clip_path, in_tc, out_tc, transition_to_next: {type, duration}}` entries.
- Hard-cut rendering via the ffmpeg **concat demuxer** with `-c copy` (stream copy / pure remux) wherever source clips share codec/profile/resolution/pixel-format/timebase and cut points are keyframe-aligned — never the ffmpeg concat *filter* and never MoviePy for final rendering, since both force a decode→re-encode round trip with no stream-copy mode.
- Transition rendering (crossfade, wipe, whip-pan-style, etc.) via ffmpeg `xfade`, scoped to only the narrow overlapping window between adjacent clips, encoded losslessly (e.g. `-crf 0`, FFV1, or ProRes 4444) with color metadata (`-colorspace`, `-color_primaries`, `-color_trc`, `-color_range`) explicitly pinned from the source via `ffprobe` — never left to ffmpeg's default colorspace guessing.
- "Match-cut" handled as an editorial choice of hard-cut point (via the concat-demuxer path), not a rendering filter.
- Duration/pacing control (target total length, per-segment trim, ordering) as an explicit, inspectable step prior to rendering.
- Export of the manifest to OpenTimelineIO (`.otio`) and, derived from that, a CMX3600 EDL (via `otio-cmx3600-adapter`) for opening the cut list in an NLE (Premiere/Resolve/Avid) for human visual confirmation — treated as a secondary, lossy-for-custom-transition-metadata export, never the primary manifest.
- A verification step distinct from visual inspection: `ffmpeg -f framemd5` comparison between source frame ranges and the corresponding output frame ranges for all stream-copied (non-transition) regions, to mechanically confirm bit-exact preservation.

**Capability 3 — Reference pack**
- A metadata-first exemplar dataset: per exemplar, canonical source URL, metadata fetched via the official YouTube Data API/oEmbed or Vimeo API/oEmbed (title, creator, duration, thumbnail, publish date), and this project's own derived analysis (the same per-signal quality scores as capability 1, plus shot-length/cut/transition-style notes) produced by manual review or a transient stream-and-discard pass that never persists the source file to disk.
- Actual footage-file retention reserved strictly for content individually verified as CC BY / CC0 / public domain / owned footage — verified per-upload, not inferred from "it won an award" or "it's a public showcase."
- A defined exemplar schema (source URL, platform, license category, per-signal + composite quality scores, award/showcase provenance if any, retrieval date) to support later "does this candidate reel score like an award-winner" comparisons.
- The pack kept strictly internal/unshared per the licensing research's risk posture (see human-signoff flags in Open Questions).

## Scope (out)

- **Capability 2 must never perform color grading, color correction, exposure/white-balance adjustment, sharpening, stabilization/warping, letterboxing, speed ramping, or any other pixel-level modification of source frames.** This is a hard non-goal, not a tunable default — every rendered frame in the final reel must be either a byte-exact stream copy of source, or (only within an explicit transition window) a lossless re-encode with color metadata pinned to match source. The `post_processing.color` / `stabilize` / `letterbox` / `auto_speed` fields present in the legacy `DJI_0355/manifest.json` precedent are explicitly excluded from this pipeline's stitching input/output contract.
- Bulk-downloading and retaining full copies of all-rights-reserved videos for the reference pack (Drone Awards / SkyPixel / Dronestagram winners, or "notable creator" YouTube/Vimeo uploads) — this is an unambiguous ToS violation on both platforms with no private/internal-use carve-out, and is out of scope for any default/automated pipeline behavior (see Tier C flag in Open Questions).
- Automating the "transient stream, analyze, discard" pattern in capability 3 beyond ad hoc/manual, human-reviewed runs — scaling it into an unattended bulk pipeline is out of scope pending legal review.
- Using YOLOv8/Ultralytics (AGPL-3.0) or pyiqa/IQA-PyTorch (PolyForm Noncommercial / NTU S-Lab / CC-BY-NC) anywhere in the default pipeline, since neither meets the permissive-license requirement — these remain excluded, not merely deprioritized.
- Deep-learning shot-boundary detection (e.g. TransNetV2) for capability 1 — drone footage is treated as one continuous take with camera motion rather than movie-style hard cuts/dissolves, so the lighter PySceneDetect approach is in scope and the added TF/PyTorch dependency is out of scope, pending evidence this assumption is wrong for real source footage.
- Rewriting or restructuring the inherited `claude-template` maintainer content (`docs/`, `promptlab/`, `copilot-studio/`, `assets/`, `dist/`, `site/`, `mkdocs.yml`, `sync-claude-template.sh`, `requirements-promptlab.txt`, template `README.md`/`CHANGELOG.md`/`TODO.md`/`REVIEW.md`/`PROMPTLAB-READINESS.md`) — that decision is a prerequisite Tier C action tracked separately, not part of this spec's Acceptance Criteria.
- Establishing `.claude/` scaffold itself, `git init`, `.gitignore` authoring, and any repo-restructuring/data-relocation (e.g. moving `example-photos-videos/` into a `data/raw/` convention) — these are Plan-phase/prerequisite actions, not part of this Spec artifact.
- Writing any application code, the Plan, or the Tasks list — this document is the Spec artifact only, per the repo's Spec-Driven Workflow.

## Acceptance Criteria

**Capability 1 — Highlight extraction**
1. Given a raw drone video file, the pipeline emits a JSON manifest containing, per selected segment: `start_time`, `end_time`, `duration`, per-signal scores (`sharpness`, `exposure`, `motion_smoothness`, `composition`), and a `composite_score` — each per-signal score normalized to a documented 0–1 (or 0–100, matching the existing manifest precedent) range with the normalization method stated in the manifest or its schema doc.
2. Segments failing any hard gate (`blackdetect`, `freezedetect`, or a configurable minimum sharpness/exposure floor) are excluded from output entirely and are distinguishable in any debug/verbose manifest from segments merely scoring low.
3. No emitted segment boundary falls strictly inside a detected scene/shot (per PySceneDetect) or strictly inside a motion-maneuver (per the motion-derivative-minima segmentation) — i.e., every emitted `start_time`/`end_time` matches a candidate boundary from Stage 1, verifiable by asserting boundary-set membership in a test.
4. Segment durations respect the configured min/max bounds (e.g. 2–15s, matching the existing manifest's 7–15s precedent as a default) in 100% of emitted segments.
5. The manifest schema is versioned (`"version": N` field, following the existing `DJI_0355/manifest.json` precedent) and does not include any `post_processing` pixel-editing fields.
6. A test suite (per Constitution rule 3) exists exercising the scoring functions against known-good/known-bad synthetic or sample inputs (e.g. a black-frame clip must hard-gate-fail; a high-Laplacian-variance clip must score above a low-variance clip).

**Capability 2 — Reel stitching**
1. Given a JSON edit manifest and source clips sharing codec/resolution/pixel-format/timebase, the rendered output's stream-copied regions are byte-exact to source, verified by matching `ffmpeg -f framemd5` output between source frame ranges and corresponding output frame ranges — this check is part of the automated test suite, not a manual/visual spot check.
2. For any transition-window region, the color metadata (`color_primaries`, `color_trc`, `color_range`, colorspace/matrix) of the rendered output, as read by `ffprobe`, exactly matches the corresponding source clip's metadata — verified programmatically, not assumed.
3. No CLI flag, config option, or code path in this capability exposes color grading, exposure/white-balance adjustment, sharpening, stabilization, letterboxing, or speed-ramping — enforced by an explicit test/lint (e.g. grepping the render module for forbidden ffmpeg filter names such as `eq`, `curves`, `colorbalance`, `unsharp`, `vidstabtransform`, `setpts` used for speed changes) rather than relying on convention alone, per the repo's "no placeholders / tool-grounded verification" constitution rules.
4. The pipeline emits both the primary JSON manifest and a derived `.otio` timeline; a CMX3600 EDL export succeeds via `otioconvert`/`otio-cmx3600-adapter` and opens without error in at least one target NLE format check (schema/structural validation is sufficient for automated CI; actual NLE round-trip is a manual acceptance step).
5. Duration/pacing parameters (target total reel length, per-clip trim) are honored within a defined tolerance (e.g. ±0.5s) of the requested total duration in test runs.

**Capability 3 — Reference pack**
1. The exemplar schema is defined and documented with at minimum these fields: `source_url`, `platform` (YouTube/Vimeo/other), `license_category` (all-rights-reserved / CC BY / CC BY-NC / CC BY-SA / CC BY-ND / CC0 / public-domain / owned), `title`, `creator`, `duration`, `retrieval_date`, per-signal quality scores matching capability 1's schema (`sharpness`, `exposure`, `motion_smoothness`, `composition`, `composite_score`), and `award_or_showcase_provenance` (nullable).
2. No exemplar record with `license_category = all-rights-reserved` has an associated locally-stored video file — verified by a schema/storage-layout check (e.g. a test asserting no file path exists under the pack's media directory for any record not tagged CC BY/CC0/public-domain/owned).
3. Metadata retrieved via the official YouTube Data API is refreshed or deleted within 30 calendar days of retrieval, per YouTube's API Services Terms — tracked via a `retrieval_date` field and a documented refresh/expiry process (automation of the refresh itself is out of scope for this spec; the field and process must exist).
4. At least one worked example (a single manually-reviewed exemplar) demonstrates the full metadata-plus-derived-analysis record end-to-end before the pattern is scaled to additional exemplars.

## Open Questions

1. **Spec granularity** — should this single combined spec be split into three per-capability specs (`.claude/specs/highlight-extraction/spec.md`, `.claude/specs/reel-stitching/spec.md`, `.claude/specs/reference-pack-curation/spec.md`) before Plan begins, as the repo-structure research recommended, given each capability has materially different acceptance criteria and risk profiles (capability 3 alone carries legal/licensing risk the other two don't)?
2. **Footage storage location/volume** — where does raw/processed footage live (local disk under a `data/raw/` convention, external drive, cloud storage/Git LFS), and what retention policy applies to intermediate artifacts (rendered highlight clips, stitched reels)? `example-photos-videos/` is already 534 MB uncompressed in a repo with no `.gitignore` and no git history yet — this must be resolved before any commit touches that directory.
3. **Reference-pack footage: metadata-only vs. any local storage at all** — confirm the default posture (metadata + source link + derived analysis only, no persisted video except individually-verified CC BY/CC0/public-domain/owned content) is acceptable, or whether the user wants a stricter metadata-only-with-zero-exceptions policy given the ambiguity the research flagged around YouTube API's "no insights into YouTube's business" clause and the risk of "transient stream, analyze, discard" starting to resemble bulk downloading at scale.
4. **GPU/CPU constraints for quality scoring** — is GPU acceleration available/required for the composition-scoring saliency models or optical-flow motion analysis at the expected footage volume, or must the pipeline run CPU-only (relevant to choosing between OpenCV's classical `saliency`/optical-flow modules, which are CPU-friendly, versus any GPU-accelerated alternative)? This also bears on whether MediaPipe's built-in object detector (a CPU-friendly, permissively-licensed alternative to YOLOv8) is sufficient for composition scoring.
5. **Recommended tech stack — proposals pending sign-off, not final decisions:**
   - Segmentation: PySceneDetect (`AdaptiveDetector`) for scene/shot boundaries.
   - Sharpness: OpenCV (`cv2.Laplacian(...).var()`).
   - Exposure: ffmpeg `signalstats`, or a NumPy/OpenCV histogram fallback.
   - Motion smoothness: either ffmpeg `vid.stab`'s `vidstabdetect` first-pass log (GPL-licensed — flagged for confirmation if GPL entanglement is a concern for this project's distribution model) or a pure-OpenCV optical-flow reimplementation (`goodFeaturesToTrack` + `calcOpticalFlowPyrLK`, following the MIT-licensed but low-activity `vidstab`/`python_video_stab` package as a reference) to avoid the GPL dependency entirely.
   - Composition: OpenCV `saliency` module (rule-of-thirds distance) plus a vendored (not pip-installed — no maintained package exists) Hough-line horizon-tilt scorer; MediaPipe's object detector as a permissively-licensed alternative to YOLOv8 (AGPL-3.0, excluded) if subject-centroid detection is needed.
   - Hard-gates: ffmpeg `blackdetect`/`freezedetect`.
   - Stitching/rendering: ffmpeg concat demuxer (`-c copy`) for hard cuts, ffmpeg `xfade` scoped to transition windows only, explicit `ffprobe`-sourced color-metadata pinning. Not MoviePy, not the ffmpeg concat filter, for final render (prototyping-only use is acceptable if the user wants a scripting convenience layer).
   - Manifest/interchange: custom JSON manifest as source of truth; OpenTimelineIO (`.otio`) + `otio-cmx3600-adapter` for NLE interchange export.
   - Reference-pack metadata retrieval: YouTube Data API v3 / YouTube oEmbed, Vimeo API / Vimeo oEmbed; `yt-dlp` restricted to `--skip-download`/metadata-only mode except for individually-verified CC/public-domain/owned content.
   - Confirm each of the above (and any substitutions) before Plan phase — none of this constitutes a locked architecture decision yet.
6. **Legal/licensing sign-off (Tier C, per the licensing research's human-signoff flags)** — before any reference-pack automation is built, get explicit confirmation on: (a) never bulk-downloading all-rights-reserved footage under any framing; (b) the boundary condition for scaling "transient stream, analyze, discard" beyond manual one-off runs; (c) how CC BY-NC exemplars may or may not be used if the reference pack's output ever touches paid/commercial work; (d) whether direct creator outreach for permission is in scope for any specific must-have exemplar; (e) confirmation the pack stays internal/unshared, since distribution shifts the risk category materially.
7. **Prerequisite sequencing** — this spec assumes `.claude/` scaffold establishment, disposition of the inherited `claude-template` maintainer content, `git init` + `.gitignore` authoring, and relocation of `example-photos-videos/` are handled as separate prerequisite actions before or alongside this feature's Plan phase — confirm that sequencing rather than folding those decisions into this spec's Acceptance Criteria.
